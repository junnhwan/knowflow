# MEMORY

## Voice
- 使用中文交流，偏务实、直接、少空话。
- 在项目和简历表述里少强调框架自带能力，多强调自己实现的系统逻辑。

## Process
- 开始任务时先读取 `context/` 与 `MEMORY.md`。
- 重要的规划、取舍和项目定位在讨论确认后要沉淀到 `docs/` 或 `context/`，方便后续连续协作。
- 若是 `KnowFlow` 的 V1 规划，优先选择可演示、可讲清楚、可写简历的 MVP，而不是摊大范围。
- 多阶段开发任务优先拆成“总计划索引 + 分 Task 文档”，这样后续修改可以精确到单个阶段，不容易偏离方向。
- 代码实现阶段要按逻辑阶段及时做 `git commit`，不要等到堆很多改动后再一次性提交。

## People
- 暂无需要长期记忆的人物信息。

## Projects
- `KnowFlow` 是一个 `Go + Eino` 的 AI 后端项目，参考 `GopherAI-v2` 的 Go/Eino 基础能力与 `ByteCoach` 的 RAG/记忆亮点。
- `KnowFlow V1` 已锁定为“面向后端面试知识运营的简历型 MVP”，主线是：文档摄取 -> 混合检索 -> Rerank -> 引用式回答 -> Redis 会话记忆 -> 知识反写 -> 索引热更新 -> 可观测性。
- `KnowFlow V1` 的核心亮点优先级：高质量 RAG、Redis 会话记忆、知识反写热更新、可观测性。
- `KnowFlow` 的“混合检索 + 融合”属于项目自身增强设计，不按 `ByteCoach` 现有代码做原样复刻。

## Output
- 规划文档放在 `docs/`，可复用的长期上下文放在 `context/`。
- 简历/项目亮点优先写系统设计、数据流、降级策略、可观测性等自己实现的部分，不把框架能力当主卖点。
- 未经过真实测量的收益数据不要直接写进正式简历。
- 对于复杂开发计划，优先输出中文，并把每个阶段拆成独立文档，在总计划中保留引用索引。
- `KnowFlow` 的正式叙事默认写成“后端面试知识运营后端”，Demo 数据、Prompt、样例问答与知识反写示例都围绕后端面试场景展开。
- `git commit` 信息默认使用中文。
