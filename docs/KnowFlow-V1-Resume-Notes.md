# KnowFlow V1 简历说明草稿

## 当前实现可支撑的项目表述

- 设计并实现面向后端面试知识运营场景的 `Go + AI` 后端，打通文档摄取、混合检索、引用式问答、知识反写与索引重建的主链路。
- 实现基于 `PgVector + pg_trgm` 的混合检索链路，包含查询预处理、`RRF` 融合、Rerank 降级与引用组装，避免把“框架调用”包装成项目亮点。
- 实现基于 Redis 的会话记忆模块，支持最近消息窗口、历史摘要压缩、TTL 生命周期与分布式锁保护，把记忆问题显式建模为状态管理问题。
- 构建轻量工具注册表，将知识检索、知识反写、文档重建索引收敛为明确业务动作，并通过 Prometheus 指标暴露 `RAG hit/miss`、`tool call`、`rerank fallback` 等关键观测点。

## 面试展开建议

- 重点讲“为什么要做混合检索而不是单路向量召回”，以及 `RRF + rerank fallback` 的质量与稳定性取舍。
- 重点讲 Redis 记忆设计：`recent + summary` 分层、压缩触发条件、锁失败时的降级写入。
- 重点讲知识反写链路为什么不能直接追加文件，而要进入结构化 `knowledge_entries`。
- 重点讲业务层如何通过本地接口封装 `Embedding / Rerank / Chat` 能力，避免代码被外部框架类型污染。

## 当前代码状态说明

- 默认运行链路使用本地 provider，便于无外部模型依赖地跑通主流程。
- 已补充可选的 `Eino` 聊天回答适配器，用于在配置真实模型时切换到 `Go + Eino` 模式。
- 当前尚未补真实 `Eino` embedding 适配器；因此如果简历里写 `Eino`，建议突出“回答编排与模型接入适配”而不是泛化成“所有 AI 能力都由 Eino 驱动”。

## 不建议直接写进简历的内容

- 未经过真实压测或线上观测的数据收益。
- “企业级平台”“多 Agent 编排”“记忆自进化”这类超出当前实现边界的表述。
- 把 `Gin`、`Eino`、`Redis`、`PgVector` 本身写成成果，而不是围绕你的设计与实现展开。
