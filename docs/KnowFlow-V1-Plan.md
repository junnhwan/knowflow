# KnowFlow V1 开发计划

## Summary

KnowFlow 的第一版定位为：**面向个人知识库场景的知识运营 Agent 后端系统**。  
目标不是做一个泛聊天平台，而是把这条主链路做深做稳：

`文档摄取 -> 混合检索 -> Rerank 精排 -> 引用式回答 -> 知识反写 -> 索引热更新`

第一版采用 **Go + Eino + Postgres/PgVector + Redis**，以纯后端优先方式推进，优先交付一个 **2-4 周内可演示、可写简历、可经受面试追问** 的 MVP。

第一版核心亮点锁定为 4 个：

- 高质量 RAG：递归切块、双路召回、Rerank、引用返回
- 分布式会话记忆：会话隔离、滑动窗口、摘要压缩、TTL
- 轻量工具 Agent：知识检索、知识入库、索引刷新
- 可观测性：检索命中率、Rerank 降级、模型耗时、工具调用成功率

## Implementation Changes

### 1. 系统边界与裁剪

- 以 `GopherAI-v2` 后端能力为参考，但第一版不保留 `图像识别 / TTS / 多功能演示页` 这类非主线模块。
- 第一版只做纯后端，不投入前端重构；保留最小接口调试与流式响应能力。
- 第一版仅服务个人知识库，不做团队协作、多租户、复杂权限系统。
- Agent 深度控制为轻量工具调用，不做复杂多 Agent 编排。

### 2. 数据与存储模型

- 使用 Postgres 作为主业务库，PgVector 作为向量检索基础。
- Redis 只承担会话记忆、短期缓存、分布式锁，不承担主向量库职责。
- 至少建立以下核心实体：
  - `documents`：文档元数据
  - `document_chunks`：切块内容、chunk 序号、来源信息、向量字段
  - `sessions`：会话信息
  - `messages`：用户消息、AI 回复、工具调用轨迹
  - `knowledge_entries`：反写知识条目及来源
- `document_chunks` 至少保留这些字段：
  - `document_id`
  - `source_name`
  - `chunk_index`
  - `content`
  - `embedding`
  - `created_at`
  - `updated_at`

### 3. 文档摄取与索引构建

- 第一版支持 `txt / md` 文档上传。
- 文档摄取流程固定为：
  - 上传文件
  - 文本抽取与规范化
  - 递归切块
  - metadata 补充
  - embedding 生成
  - chunk 与向量落库
- 切块策略采用递归切分，不做简单整篇入库。
- 切块参数第一版固定，避免过早开放配置：
  - `chunk_size`：约 `600-800` 字符
  - `chunk_overlap`：约 `120-200` 字符
- 文档更新时，采用“按文档全量重建 chunk”策略，不做复杂增量切块。
- 文档删除时，必须同步删除对应 chunk 与向量记录。

### 4. 检索链路

- 检索链路采用混合检索，而不是单路向量检索：
  - 向量召回
  - 关键词召回
- 第一版融合策略采用**简单可解释方案**，优先使用：
  - `RRF`，或
  - 归一化分数加权融合
- 在融合后的候选集上执行 Rerank 精排：
  - 粗排召回较大候选集
  - 精排返回最终 `TopN`
- 对查询增加预处理：
  - 去除多余标点
  - 空白规范化
  - 中文停用词过滤
- 回答生成必须包含引用能力：
  - 返回引用片段
  - 返回来源文档名
  - 返回 chunk 标识
- 检索失败或质量不足时的行为固定为：
  - 无结果时明确拒答
  - Rerank 失败时回退到粗排结果
  - 不允许“无依据编造”

### 5. 会话记忆系统

- 使用 Redis 建立会话记忆，主键采用：
  - `user_id + session_id`
- 记忆分为两层：
  - 最近 N 轮完整对话
  - 历史摘要记忆
- 触发压缩条件固定为二选一满足即可：
  - 消息条数超过阈值
  - 估算 token 超过阈值
- 压缩策略固定为：
  - 保留最近若干轮完整消息
  - 更早历史合并为摘要块
  - 压缩失败时退化为仅保留最近若干轮
- Redis 中需实现：
  - TTL 生命周期管理
  - 分布式锁避免并发覆盖
  - 原子更新或等价的幂等更新策略
- 第一版记忆目标是“维持长对话连贯性并控制上下文成本”，不做长期人格记忆。

### 6. 轻量工具 Agent

- 第一版工具集合固定为 3 类：
  - `retrieve_knowledge`
  - `upsert_knowledge`
  - `refresh_document_index`
- 工具调用链路要求：
  - 结构化参数
  - 基础校验
  - 超时控制
  - 执行日志
  - 失败回退
- “知识自更新”在实现上拆成明确业务链路：
  - 用户提交高价值内容
  - 系统抽取结构化知识条目
  - 落业务库
  - 异步或同步触发建索引
- 第一版不做复杂审核流；如需要，预留状态字段即可。

### 7. API 设计

第一版稳定以下后端 API：

- `POST /api/kb/documents`
  - 上传文档并完成入库/建索引
- `POST /api/chat/query`
  - 基于 session 的知识问答
- `GET /api/chat/sessions`
  - 获取用户会话列表
- `POST /api/kb/knowledge`
  - 提交知识反写/知识更新
- `POST /api/kb/reindex`
  - 手动触发指定文档重建索引

`POST /api/chat/query` 返回体至少包含：

- `session_id`
- `answer`
- `citations`
- `tool_traces`（可选）
- `retrieval_meta`（可选，包含召回数量、是否触发降级）

### 8. 可观测性

- 所有请求统一携带：
  - `request_id`
  - `user_id`
  - `session_id`
- Prometheus 采集至少这些指标：
  - LLM 调用耗时
  - LLM 成功率/失败率
  - RAG hit/miss
  - Rerank fallback 次数
  - 工具调用成功率
  - 会话数、活跃用户数
- Grafana 只做主链路看板，不追求大而全。
- 日志必须能串起：
  - 请求进入
  - 检索链路
  - 模型调用
  - 工具执行
  - 响应输出

## Delivery Phases

### Phase 1：底座搭建

- 建立 Postgres + PgVector + Redis 运行环境
- 搭建核心数据模型
- 打通文档上传、解析、切块、向量入库
- 提供最小可用检索接口

### Phase 2：高质量 RAG

- 实现向量召回
- 实现关键词召回
- 实现融合策略
- 接入 Rerank
- 增加引用返回、拒答策略和降级逻辑

### Phase 3：记忆与工具 Agent

- Redis 会话记忆落地
- 滑动窗口 + 历史摘要压缩
- 工具调用链路落地
- 知识反写与索引热更新打通

### Phase 4：监控与交付

- Prometheus 指标接入
- Grafana 面板搭建
- 结构化日志完善
- 输出可用于简历和演示的项目描述

## Test Plan

- 文档摄取测试：
  - `txt / md` 可正常上传、切块、入库、重建
  - 文档更新后旧 chunk 不残留
  - 删除文档后索引记录同步删除
- 检索链路测试：
  - 向量召回、关键词召回、融合、Rerank 可分别验证
  - Rerank 失败时能正确降级
  - 无命中场景明确拒答
  - 回答中引用信息与来源正确
- 记忆系统测试：
  - 多 session 隔离正确
  - 压缩触发条件正确
  - 压缩失败降级正确
  - Redis 锁下并发写不会污染上下文
- 工具链路测试：
  - 检索工具、入库工具、重建工具参数校验正确
  - 工具失败时响应可解释，不中断主链路
- 可观测性测试：
  - 指标可采集
  - `request_id / user_id / session_id` 能贯穿日志
- 流式响应测试：
  - SSE 输出不中断
  - 长回答可分段返回
  - 会话历史可持续追问

## Resume Positioning

简历中不主写“Eino 带来的能力”，而主写“系统逻辑与实现链路”。

推荐提炼为 4 条亮点：

- 设计并实现知识运营 Agent 后端，支持文档摄取、混合检索、Rerank 精排与引用式回答，提升私有知识问答的准确性与可解释性
- 实现基于 Redis 的会话记忆系统，通过滑动窗口与历史摘要压缩控制上下文长度，降低长对话场景下的模型调用成本
- 设计知识反写与热更新机制，支持高价值问答沉淀为知识条目，并完成结构化入库与索引更新
- 构建 AI 链路可观测体系，监控模型调用耗时、检索命中率、Rerank 降级与工具调用成功率，提升问题定位与系统优化效率

## Assumptions

- 项目名称为 `KnowFlow`。
- 第一版采用纯后端优先策略。
- 第一版目标周期为 `2-4 周`，以可演示、可写简历、可讲清楚为标准。
- 第一版仅支持 `txt / md` 文档。
- 第一版不做企业多租户、不做复杂权限、不做前端重构。
- 第一版 Agent 仅为轻量工具调用，不做复杂多 Agent 工作流。
- 所有量化收益在没有真实测试数据前，不写入正式简历。
