# KnowFlow V1 设计记录

日期：2026-04-02

## 1. 背景与目标

`KnowFlow` 的起点不是“再做一个 AI 聊天项目”，而是做一个更适合当前阶段的 `Go + AI` 代表项目：

- 能作为 Java 后端转 Go 后端的核心作品；
- 能写进简历，而且经得住后端面试追问；
- 能把 AI 项目的亮点落到真实的系统设计、状态管理和数据链路上；
- 能在 2-4 周内做出一个可演示、可继续演进的 V1。

这次规划综合了两个参考项目：

- `D:\dev\go_proj\GopherAI\GopherAI-v2`
- `D:\dev\learn_proj\ByteCoach`

最终目标不是把几个项目机械拼接，而是抽出最适合 `Go + Eino` 落地、也最适合简历表达的部分，形成 `KnowFlow` 自己的项目闭环。

## 2. 参考项目拆解结论

### 2.1 GopherAI-v2 的价值

`GopherAI-v2` 更适合作为 `KnowFlow` 的 Go 技术底座参考，而不是简历亮点模板。

可借鉴的部分：

- Go 项目组织方式与基础工程结构；
- `Eino` 接入经验；
- 聊天接口设计与 `SSE` 流式输出；
- 基础 RAG 的接入范式；
- 会话接口、模型调用、流式响应等后端链路形态。

不建议直接继承的部分：

- 目前 RAG 仍偏单路检索；
- 文档切块、检索质量、会话记忆还不够深；
- 现有会话状态更偏进程内管理，不适合作为 `KnowFlow` 的长期目标。

### 2.2 ByteCoach 的价值

`ByteCoach` 更适合作为“亮点方向参考”。

当前源码里最有参考价值的部分主要是：

- 递归切块；
- 查询预处理；
- 向量召回 + `Rerank` 精排；
- Redis 会话记忆隔离；
- 滑动窗口 + 历史摘要压缩；
- TTL、分布式锁、并发更新保护；
- 知识反写与索引热更新；
- Prometheus / Grafana 监控意识。

需要特别区分的一点：

- `ByteCoach` 对外亮点描述里提到过“双路召回 + RRF”，但当前源码里更明确落地的是“向量召回 + 查询预处理 + Rerank”。
- 因此 `KnowFlow` 若做“混合检索 + 融合”，这应该被视为 `KnowFlow` 的明确设计目标，而不是对 `ByteCoach` 现有实现的直接复刻。

### 2.3 ragent 的价值

`ragent` 更适合作为“工程拆分模式参考”。

可借鉴的部分：

- 检索通道接口 + 后处理器链的拆分思路；
- 检索、记忆、Trace 的职责边界；
- 用清晰服务接口包住基础设施实现，而不是让业务层直接依赖外部组件。

不建议直接带入 `V1` 的部分：

- 意图树与复杂问题重写链路；
- 多模型路由、熔断、首包探测；
- 后台管理、大量线程池、限流排队与平台化配置体系。

### 2.3 不建议复刻的部分

以下内容不适合作为 `KnowFlow V1` 的核心方向：

- “企业级智能 Agent 平台”这种过大的叙事；
- 复杂多 Agent 编排；
- 把 Guardrail 当成主要卖点；
- 把框架提供的能力直接当作自己的项目亮点；
- 没有真实测试前就写进简历的收益数字。

## 3. 可选方向与最终选择

### 3.1 快速复刻型

以 `GopherAI-v2` 为骨架，快速改成一个能跑的 `Go + Eino` AI 项目。

优点：

- 起步快；
- 开发阻力小；
- 能迅速搭出最小功能链路。

问题：

- 容易像“改造 demo”；
- 项目辨识度有限；
- 面试时不容易突出自己真正实现的系统逻辑。

### 3.2 简历型 MVP

围绕主链路只做 4 个硬亮点：

- 高质量 RAG；
- Redis 会话记忆；
- 知识反写热更新；
- 可观测性。

优点：

- 范围可控；
- 最适合写简历和面试展开；
- 能把每个模块讲成真正的后端系统设计。

这是 `KnowFlow V1` 的最终选择。

### 3.3 学习型全家桶

尽可能覆盖 ByteCoach 的更多方向，例如 Guardrail、多工具、更多编排、更多平台能力。

优点：

- 接触面广；
- 学习内容多。

问题：

- 容易摊大；
- 很多地方会停留在“用了框架”的层面；
- V1 难以形成一个完整、扎实、可讲清楚的后端闭环。

## 4. 项目定位与边界

### 4.1 项目定位

`KnowFlow` 的定位是：

> 面向后端面试知识运营场景的知识运营 AI 后端。

它不追求“泛聊天平台”，而是把一条知识链路做深做稳：

`文档摄取 -> 混合检索 -> Rerank -> 引用式回答 -> Redis 会话记忆 -> 知识反写 -> 索引热更新 -> 可观测性`

### 4.2 V1 边界

`KnowFlow V1` 明确只做以下范围：

- 纯后端优先，提供 HTTP API 与 `SSE` 流式输出；
- 文档类型先只支持 `txt / md`；
- 单用户 / 个人知识库场景优先；
- Agent 只保留轻量工具调用；
- 主存储使用 `Postgres + PgVector`；
- 会话与缓存使用 `Redis`；
- `Eino` 作为编排和模型接入工具，而不是项目卖点本身。
- Demo 数据、Prompt、样例问答、知识反写示例默认围绕后端面试题、项目复盘和八股沉淀展开。

### 4.3 V1 不做项

- 不做前端重构；
- 不做 PDF/Word/网页抓取；
- 不做多租户与复杂权限；
- 不做复杂审批流；
- 不做复杂多 Agent 工作流；
- 不把 Guardrail / MCP 生态扩展作为 V1 主卖点。
- 不引入 `ragent` 风格的意图树、多模型路由、后台管理、队列式限流等平台化能力。

## 5. 核心亮点设计

### 5.1 高质量 RAG

这是 `KnowFlow` 的第一主亮点。

V1 目标：

- 文档规范化与递归切块；
- `PgVector` 语义召回；
- `pg_trgm` 关键词召回；
- `RRF` 融合策略；
- `Rerank` 精排；
- 引用式回答；
- 无依据拒答；
- 检索与精排失败时可解释降级。

要点不是“接了向量库”，而是整条检索链路具备质量控制能力。

### 5.2 Redis 会话记忆

这是 `KnowFlow` 的第二主亮点。

V1 目标：

- 会话隔离；
- 最近消息窗口；
- 历史摘要压缩；
- TTL 生命周期；
- 分布式锁；
- 并发写保护；
- 压缩失败降级。

重点要讲成一个状态管理问题，而不是“框架自动帮我记住上下文”。

### 5.3 知识反写与索引热更新

这是 `KnowFlow` 的第三主亮点。

V1 目标：

- 用户提交高价值知识；
- 系统抽取并保存为结构化知识条目；
- 更新业务库；
- 只刷新受影响的文档或知识片段索引；
- 让知识库具备持续生长能力。

这里要强调“知识沉淀链路”而不是抽象的“记忆自进化”。

### 5.4 可观测性

这是 `KnowFlow` 工程完成度的重要体现。

V1 目标：

- 统一 `request_id / user_id / session_id`；
- 记录模型调用耗时与成功率；
- 记录 `RAG hit/miss`；
- 记录 `Rerank fallback`；
- 记录工具调用成功率；
- 输出可用于排障和调优的日志与指标。

## 6. 模块拆分

`KnowFlow V1` 建议按业务闭环拆模块，而不是平铺成一层层传统目录。

### 6.1 Document Ingestion

职责：

- 上传文档；
- 文本提取与规范化；
- 递归切块；
- metadata 补充；
- embedding 生成；
- chunk 入库。

### 6.2 Retrieval Pipeline

职责：

- 查询预处理；
- 向量召回；
- 关键词召回；
- 融合；
- `Rerank`；
- 引用组装；
- 拒答与降级。

### 6.3 Session Memory

职责：

- 加载最近消息窗口；
- 管理历史摘要；
- 触发压缩；
- 控制上下文长度；
- 使用 Redis 锁避免并发污染；
- TTL 生命周期管理。

### 6.4 Tool Agent

V1 工具仅保留 3 类：

- `retrieve_knowledge`
- `upsert_knowledge`
- `refresh_document_index`

目标是把自然语言意图映射到明确业务动作，而不是追求工具数量，也不是把框架自带 Function Calling 当成卖点。

### 6.5 Chat Orchestration

职责：

- 加载会话上下文；
- 串联检索结果、工具结果和模型生成；
- 输出普通响应或流式响应；
- 保存消息轨迹与工具轨迹。

### 6.6 Observability

职责：

- 结构化日志；
- Prometheus 指标；
- 错误降级记录；
- 请求链路追踪字段。

## 7. 数据模型设计

V1 至少需要以下核心表：

### 7.1 `documents`

存文档元信息，建议包含：

- `id`
- `source_name`
- `status`
- `content_hash`
- `created_at`
- `updated_at`

### 7.2 `document_chunks`

存切块与向量索引主数据，建议包含：

- `id`
- `document_id`
- `chunk_index`
- `content`
- `embedding`
- `token_count`
- `created_at`
- `updated_at`

### 7.3 `sessions`

存会话基础信息，建议包含：

- `id`
- `user_id`
- `title`
- `last_active_at`
- `created_at`

### 7.4 `messages`

存消息轨迹，建议包含：

- `id`
- `session_id`
- `role`
- `content`
- `tool_name`
- `tool_input`
- `tool_output`
- `created_at`

### 7.5 `knowledge_entries`

存对话沉淀出的结构化知识条目，建议包含：

- `id`
- `session_id`
- `source_type`
- `content`
- `status`
- `created_at`

一个需要长期坚持的设计原则：

- `document_chunks` 是检索主表；
- `knowledge_entries` 是业务知识沉淀表；
- 两者生命周期不同，不建议混成一张表。
- 不采用“直接往 markdown 文件 append 内容”来代替结构化知识反写。

## 8. 核心请求链路

### 8.1 文档摄取链路

`POST /api/kb/documents`

处理步骤：

1. 接收文档并校验类型；
2. 提取文本并做基础规范化；
3. 递归切块；
4. 写入 `documents` 与 `document_chunks`；
5. 生成 embedding 并入 `PgVector`；
6. 返回入库结果与文档标识。

### 8.2 问答链路

`POST /api/chat/query`

处理步骤：

1. 根据 `user_id + session_id` 加载 Redis 会话记忆；
2. 获取最近消息窗口与历史摘要；
3. 对当前问题做查询预处理；
4. 执行向量召回 + 关键词召回；
5. 对候选结果做融合与 `Rerank`；
6. 若结果不足则进入拒答或降级；
7. 组装引用片段；
8. 将检索结果、记忆与用户问题交给模型；
9. 返回回答、引用信息、工具轨迹、检索元信息；
10. 更新消息历史与记忆状态。

补充稳定 API：

- `POST /api/chat/query/stream`
- `GET /api/chat/sessions`
- `GET /api/chat/sessions/{session_id}/messages`

### 8.3 知识反写链路

`POST /api/kb/knowledge`

处理步骤：

1. 接收高价值问答或知识片段；
2. 抽取结构化内容；
3. 写入 `knowledge_entries`；
4. 触发对应索引更新；
5. 返回更新结果。

### 8.4 手动重建链路

`POST /api/kb/reindex`

处理步骤：

1. 指定文档或知识条目范围；
2. 删除旧索引；
3. 重新切块与向量化；
4. 返回重建状态与耗时信息。

## 9. 降级与异常策略

V1 必须从一开始就保留这些“后端思维”：

### 9.1 检索链路

- 向量召回无结果时返回明确拒答；
- 关键词召回失败不影响主流程；
- `Rerank` 失败时回退到粗排结果；
- 引用不足时不允许凭空编造“有依据”的回答。

### 9.2 记忆链路

- 超过消息数或 token 阈值时触发压缩；
- 压缩失败时只保留最近若干轮完整消息；
- Redis 锁竞争失败时需要有降级写入策略；
- 会话状态更新尽量保证原子性或幂等性。

### 9.3 工具链路

- 参数必须结构化校验；
- 工具要有超时控制；
- 工具失败不能直接拖垮主链路；
- 失败信息要能被日志与监控捕获。

## 10. 测试与验证思路

V1 的测试重点不是覆盖率数字，而是关键链路的稳定性。

建议优先验证：

- 文档上传、切块、入库、删除、重建；
- 向量召回、关键词召回、融合、`Rerank`、拒答、引用返回；
- 多会话隔离；
- 记忆压缩触发与降级；
- Redis 锁下的并发写保护；
- 工具调用参数校验与失败回退；
- 指标采集与日志链路；
- `SSE` 长响应不中断。

## 11. 简历表达原则

`KnowFlow` 的简历描述不应该把重点放在 `Eino`、`PgVector`、`Redis` 这些名词本身，而应该放在自己设计和实现的链路上。

建议突出：

- 高质量 RAG 检索链路；
- Redis 会话记忆与上下文压缩；
- 知识反写与热更新；
- AI 链路可观测性；
- 降级与稳定性设计。
- 面向后端面试知识运营场景的知识沉淀闭环。

不建议直接写：

- “框架自带工具调用”
- “企业级平台”
- “多 Agent 编排”
- “收益提升 xx%”但没有真实测量

## 12. 当前结论

`KnowFlow V1` 已明确采用“简历型 MVP”路线。

第一阶段不追求做大，而是先把以下 4 个亮点做实：

- 高质量 RAG
- Redis 会话记忆
- 知识反写热更新
- 可观测性

后续流程建议：

1. 以本设计记录为基础，补齐正式实现计划；
2. 再进入编码与环境搭建阶段；
3. 每完成一个主链路模块，就同步更新文档与简历表述。
